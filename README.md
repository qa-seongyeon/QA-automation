사용자의 흐름을 따라가는 UI/API E2E 테스트와,<br>
기능 검증 위주의 리그리에션 테스트, 서버 부하 검증까지 고려한 자동화 프로젝트
# AutomationExercise 자동화 테스트
---
## 프로젝트 소개

사이트 URL : [AutomationExercise](https://automationexercise.com) <br>
UI 자동화와, API 검증을 한 프로젝트에 담기 위해 해당 사이트로 선택하였습니다.<br>
Playwright와 Pytest를 사용하여 Python언어로 작성되었습니다.<br>
<br>
해당 프로젝트는 자동화 전용 테스트 케이스를 먼저 설계한 후,<br> 
리스크 기반, 중요도가 높은 시나리오를 선별하여 자동화하였습니다.<br>

로그인은 모든 서비스의 공통 기능임과 동시에 서비스의 시작점임으로,<br>
최우선 순위로 자동화를 설계하였습니다.<br>
현재까지 로그인 기능만을 포함하고 있지만<br>
핵심 비지니스 로직인 상품 검증, 장바구니, 결제 시나리오를 순차적으로 추가 구현 예정입니다.

```

│   └── data_utils.py
│
├── pageobject/         # Page Object Model
│   └── login_page.py
│

│   ├── conftest.py     # 공통 fixture
│   ├── ui_e2e/            # 사용자 여정 기반 UI 테스트
│   ├── api_e2e/           # API 조합 테스트
│   └── regression/        # 기능 검증 테스트

│
└── pytest.ini
```

## 사용 기술

* Python 3
* Playwright (sync API) — UI 자동화 및 API 테스트
* Pytest — fixture, parametrize, marker 기반 테스트 구성
* Allure — 테스트 리포트
* Locust — 부하 테스트
* python-dotenv — 테스트 계정/환경 정보 관리

## 테스트 설계 근거
확장성과 유지보수성을 위해 tests → PageObject → API → settings 계층 구조로 설계하였습니다.<br>
tests 테스트 코드가 UI 요소, 비즈니스 로직, 환경 설정에 직접 의존하지 않도록<br>
UI 변경은 Page Object에서, API 변경은 API 계층에서, 환경 변경은 settings에서만 수정되도록<br>
각 계층의 책임을 분리하였습니다.<br>
<br>
E2E TEST는 서로 다른 레이어에서 데이터/상태 전달 과정을 검증하기 때문에<br>
UI와 API를 구분하여 설계하였습니다.<br>
두 레이어를 구분하여 테스트를 설계함으로써<br>
서로가 놓치는 결함 영역을 상호 보완하도록 하였습니다.<br>
<br>
TC는 품질 특성에 따른 리스크를 정의하고, 그에 따른 세부 리스크 항목을 리스트업하여<br>
결함을 사전에 방지할 수 있도록 작성 하였습니다.<br>

## 테스트 목적

* UI E2E TEST:<br>
  E2E에서는 화면 기능 하나하나를 검증하지 않습니다<br>
  화면들을 거쳐서 최종적으로 사용자의 흐름이 완결되는가를 기준으로 작성하였습니다.<br>
  메인 페이지에서 로그인을 시도하는 케이스<br>
  장바구니 결제 유도 속에서 로그인으로 이동하는 케이스를 다뤘습니다.
  <br>
  
* API E2E TEST:<br>
  여러 API 들이 조합됐을 때 사용자의 비지니스 목적이 달성 되는가를 확인하기 위해 설계하였습니다.
  <br>
  
* Regration TEST:<br>
  화면 내 개별 단위 기능 및 예외케이스는 해당 테스트 모듈에서 검증합니다.<br>
* 동시에 보안 리스크를 고려한 SQL/XSS인젝션 테스트를 포함하도록 하였습니다.
  <br>
 
* load_tests:<br>
  사용자의 트래픽이 몰릴 것으로 예상되는 상품 조회 기능에서<br>
  서버의 순수 처리 능력을 확인하기 위하여 UI단이 아닌 API 직접 호출하여<br>
  사용자의 요청을 안정적으로 처리할 수 있는가를 확인하고자 하였습니다.

## 구현하면서 신경 쓴 부분

### 1. Page Object Model 적용

유지보수와 확장성을 위해 수정 지점을 한 곳으로 모을 수 있도록 페이지 오브젝트와<br>
테스트 시나리오(tests)를 분리하였습니다.<br>
화면 별 UI 요소와 사용자 플로우를 정의함으로써 <br>
UI 변경 시 테스트 시나리오에는 영향을 끼치지 않도록 설계하였습니다.

### 2.Pytest Marker SMOKE TEST 활용 + CI 파이프라인 연동

별도의 smoke test 모듈을 생성하지 않고,<br>
소수의 핵심 시나리오에만 `smoke` Marker를 부여하여<br>
GitHub Actions으로 push 시 스모크 테스트가 자동 실행되도록 CI 파이프라인에 연동하였습니다.<br>

### 3. parametrize와 TC 넘버 매핑

로그인 리그레이션 테스트 중 실패 시나리오의 경우<br>
동일한 검증 로직을 다양한 입력값으로 확인하기 위해 ⁠parametrize⁠로 통합하여 구현했습니다.<br>
또한, 테스트 실패 시 실패 지점과 테스트 커버리지를 직관적으로 파악하기 위해<br>
각 테스트 함수마다 TC 번호를 매핑하여 추적성을 높였습니다.<br>

### 4. fixture 기반 테스트

API 테스트 시 데이터 충돌을 막기 위해 ⁠account_data⁠ fixture를 활용했습니다.<br>
매 테스트마다 UUID 기반의 일회성 계정을 생성하여 실행하고<br>
테스트가 끝나면 Teardown 로직을 통해 생성된 계정은 자동으로 삭제되도록 하였습니다.<br>
이를 통해 테스트 간의 독립성을 보장하고, 몇 번을 반복 실행해도 데이터 무결성을 보장하도록 하였습니다.

### 5. 테스트 안정성 확보 (광고 차단)

대상 사이트의 광고 스크립트가 테스트 실행을 방해하는 문제가 있어, <br>
`page.route`로 광고 도메인 요청을 차단하는 fixture를 구성하였습니다.

### 6. 환경/계정 정보 분리

테스트 계정과 비밀번호를 `.env` 파일로 분리하였으며<br>
데모사이트는 모두 운영 환경 검증이지만,<br>
현업에서 처럼 `TEST_ENV` 환경변수로 staging/prod 대상 환경을 전환할 수 있도록 설계하였습니다.<br>
3개 이상인 다중 환경이라면 확장성을 고려하여 딕셔너리 형태로 설계하겠습니다.

### 7. 테스트 설계로 비공식 API의 리스크 보완  

장바구니 사전 조건 세팅에 활용한 내부 ⁠add_to_cart⁠ 엔드포인트는 공식 문서화되지 않은 API이므로,<br>
상태 코드 200 만을 성공 조건으로 보지 않고<br>
장바구니 화면의 상품 행 개수를 다시 확인 방식으로 이중 검증하도록 하였습니다.<br>
