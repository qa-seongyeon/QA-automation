사용자의 흐름을 따라가는 UI/API E2E 테스트와,<br>
기능 검증 위주의 리그리에션 테스트, 서버 부하 검증까지 고려한 로그인 자동화 프로젝트
# AutomationExercise 로그인 자동화 테스트
---
## 프로젝트 소개

사이트 URL : [AutomationExercise](https://automationexercise.com) <br>
UI 자동화와, API 검증을 한 프로젝트에 담기 위해 해당 사이트로 선택하였습니다.<br>
Playwright와 Pytest를 사용하여 Python언어로 작성되었습니다.<br>
<br>
해당 프로젝트는 자동화 전용 테스트 케이스를 먼저 설계한 후,<br> 
리스크 기반, 중요도가 높은 시나리오를 선별하여 자동화하였습니다.<br>
<br>
테스트의 성격에 따라 UI E2E/ API E2E, Regression, 부하 테스트로 계층을 분리하여,<br>
테스트 실패 시 어느 레벨에서 문제가 발생했는지 빠르게 파악할 수 있는 구조로 설계하였습니다.

---
## 프로젝트 구조

```
QA_DEMO/
├── api/                # API 요청 함수 계층
│   └── login_api.py
│
├── config/             # 테스트 환경 구분 및 계정 설정
│
│   └── settings.py
├── pageobject/         # Page Object Model
│   └── login_page.py
│
├── tests/              # 테스트 시나리오 모음
│   ├── conftest.py     # 공통 fixture
│   ├── ui_e2e/            # 사용자 여정 기반 UI 테스트
│   ├── api_e2e/           # API 레벨 계정/로그인 테스트
│   └── regression/        # 로그인 실패/성공 회귀 테스트
│
├── load_tests/         # Locust 부하 테스트
└── pytest.ini
```

## 사용 기술

* Python 3
* Playwright (sync API) — UI 자동화 및 API 테스트
* Pytest — fixture, parametrize, marker 기반 테스트 구성
* Allure — 테스트 리포트
* Locust — 부하 테스트
* python-dotenv — 테스트 계정/환경 정보 관리

## 테스트 설계 근거

로그인은 모든 서비스의 공통 기능임과 동시에 서비스의 시작점으로,<br>
중요도가 가장 높기 때문에최우선 순위로 TC를 설계하였습니다.

품질 특성에 따른 리스크를 정의하고, 그에 따른 세부 리스크 항목을 리스트업하여<br>
결함을 사전에 방지할 수 있는 TC를 작성하고자 하였습니다.

E2E TEST는 UI와 API를 구분하여 설계하였습니다.<br>
서로 다른 레이어에서 데이터/상태 전달 과정을 검증하기 때문에,<br>
두 레이어를 구분하여 테스트를 설계함으로써<br>
서로가 놓치는 결함 영역을 상호 보완하도록 하였습니다.

## 테스트 목적

* UI E2E TEST:<br>
  로그인 기능이 실제 사용자 여정인 메인페이지 진입, <br>
  장바구니 결제 유도 속에서 정상 동작하는지 확인하고자 하였습니다.
  <br>
  
* API E2E TEST:<br>
  여러 API 들이 조합됐을 때 사용자의 비지니스 목적이 달성 되는가를 확인하기 위해 설계하였습니다.
  <br>
  
* Regration TEST:<br>
  로그인 기능 변경 시 발생할 수 있는 결함, 영향 범위를 모두 커버할 수 있도록 기능 중점으로 TC를 작성하였습니다.
* 동시에 보안 리스크를 고려한 SQL/XSS인젝션 테스트를 포함하도록 하였습니다.
  <br>
 
* load_tests:<br>
  사용자의 트래픽이 몰릴 것으로 예상되는 상품 조회 기능에서 서버의 순수 처리 능력을 확인하기 위하여<br>
  UI단이 아닌 API 직접 호출하여 사용자의 요청을 안정적으로 처리할 수 있는가를 확인하고자 하였습니다.

## 구현하면서 신경 쓴 부분

### 1. Page Object Model 적용

유지보수와 확장성을 위해 수정 지점을 한 곳으로 모을 수 있도록 페이지 오브젝트와 <br>
테스트 시나리오(tests)를 분리하였습니다.<br>
화면 별 UI 요소와 사용자 플로우를 정의함으로써 <br>
UI 변경 시 테스트 시나리오에는 영향을 끼치지 않도록 구조화 하였습니다.

### 2. parametrize와 TC 넘버 매핑

리그레이션 테스트의 로그인 실패 케이스의<br>
같은 로직을 여러 입력값으로 검증하기 위해 <br>
입력값 검증은 parametrize로 통합하였습니다.<br>
또한 테스트 실패 추적과, 테스트 커버리지 파악을 위해<br>
각 테스트 항목마다 TC 번호를 매칭하였습니다. 

### 3. fixture 기반 테스트 격리와 정리

API 테스트에서 사용하는 계정은 `account_data` fixture가 매 테스트마다 uuid 기반의 고유 이메일로 생성하고,<br>
테스트 종료 후 teardown에서 잔여 계정을 삭제합니다. <br>
테스트 간 데이터 간섭을 차단하고 반복 실행해도 테스트가 오염되지 않도록 방지하였습니다.

### 4. 테스트 안정성 확보 (광고 차단)

대상 사이트의 광고 스크립트가 테스트 실행을 방해하는 문제가 있어, <br>
`page.route`로 광고 도메인 요청을 차단하는 fixture를 구성하였습니다.

### 5. 환경/계정 정보 분리

테스트 계정과 비밀번호를 `.env` 파일에서 로드하도록 분리했으며<br>
데모사이트는 모두 운영 환경 검증이지만,<br>
현업에서 처럼 `TEST_ENV` 환경변수로 staging/prod 대상 환경을 전환할 수 있도록 설계하였습니다.

### 6. 문서화되지 않은 결함도 대비한 테스트 설계

장바구니 사전 조건 세팅에 사용한 내부 `add_to_cart` 엔드포인트는 공식 문서화된 API가 아니기 때문에<br>
응답 상태코드 "200"만 믿지 않고<br>
장바구니 화면의 상품 행 개수를 다시 확인 방식으로 이중 검증할 수 있도록 하였습니다.
